# smcode 第 1 章：交互式对话 Spec

## 背景

smcode 是一个运行在终端的代码编写助手（Coding Agent），目标是为开发者提供类似 Claude Code 的交互式体验。当前项目仅包含最基础的命令行脚手架，启动后只输出问候语，尚未接入任何大模型能力。

本章节聚焦第一个可交付里程碑：让 smcode 进入终端后呈现一个交互式对话界面，用户能够输入问题，助手调用大模型 API 并把回答以流式方式逐字呈现；同时支持多轮对话，使 AI 能基于历史上下文继续回应。若模型返回思考过程（reasoning/thinking），TUI 同样会将其展示出来。

## 目标

- 用户在终端执行 smcode 后，进入一个全屏或伪全屏的交互式对话界面。
- 用户在输入区键入问题并提交，系统把该消息追加到当前对话历史。
- 系统根据配置文件选定的模型和请求地址，向 OpenAI 兼容的大模型 API 发起请求。
- 系统把模型返回的文本以流式方式逐字/逐段渲染到对话区，而不是等全部内容返回后再一次性显示。
- 系统把模型返回的思考过程（reasoning_content / thinking）以流式方式展示在最终答案之前或旁侧。
- 系统支持 OpenAI 风格的聊天补全协议。
- Provider 层被抽象成统一接口，后续新增后端时不需要改动 TUI 或对话流程。
- 配置信息使用 YAML 文件管理，核心字段包括协议类型、模型名称、请求地址和认证密钥。

## 功能需求

- **F1 启动交互式 TUI**：用户在终端启动应用后，进入一个交互式对话界面，界面至少包含对话展示区与用户输入区。
- **F2 用户输入与提交**：用户可在输入区键入多行或单行问题，通过提交快捷键将问题发送到系统，并在对话展示区看到自己发送的消息。
- **F3 多轮对话历史**：系统在当前会话中维护用户与助手的消息历史，每次调用模型时把完整历史作为上下文传入。
- **F4 流式响应渲染**：系统以 Server-Sent Events（SSE）方式接收模型返回的文本片段，并在对话展示区按接收顺序逐段追加显示，而不是等全部内容返回后再一次性展示。
- **F5 Provider 统一抽象**：Provider 层提供统一的对话接口，初始实现覆盖 OpenAI 风格的聊天补全协议；新增后端时只需扩展 Provider 实现，无需改动 TUI 或对话编排。
- **F6 思考过程展示**：当 OpenAI 兼容后端返回 `reasoning_content` 或类似思考字段时，系统在 TUI 中将思考过程与最终答案分开展示，并以流式方式逐段更新。
- **F7 Token 消耗展示**：系统在 TUI 底部状态栏展示当前会话累计的 token 消耗（提示 token、生成 token、总计）。
- **F8 YAML 配置管理**：系统从 YAML 配置文件中读取供应商信息，核心字段包括 `protocol`、`model`、`base_url`、`api_key`。
- **F9 配置选择**：用户能够通过命令行参数或环境变量指定使用的配置文件路径或配置项名称；未指定时使用默认路径。

## 非功能需求

- **N1 流式响应性能**：从用户提交问题到界面开始显示第一个响应片段的延迟应尽可能低，避免等待完整响应。
- **N2 Provider 可扩展性**：新增一种后端协议时，只需实现 Provider 接口并在配置中注册，不需要修改 TUI 或对话编排逻辑。
- **N3 密钥安全**：API 密钥只通过配置文件或环境变量传入，不直接写入源代码，也不在不安全的日志中打印。
- **N4 错误可读性**：当配置文件缺失、字段非法或模型 API 返回错误时，界面向用户展示可理解的错误信息，而不是原始堆栈。
- **N5 可测试性**：Provider 接口与 TUI 渲染逻辑之间职责清晰，便于在单元测试中注入模拟 Provider。
- **N6 技术栈一致**：实现使用项目已选定的 TypeScript / Node.js 运行时，与现有构建流程和类型检查配置保持一致。

## 不做的事

- 本章不接入 Claude/Anthropic 或其他非 OpenAI 格式的后端。
- 本章不实现跨会话的持久化对话历史；会话结束或应用退出后，历史记录不保留。
- 本章不实现工具调用（tool use）、文件读写、代码执行等 Agent 能力；TUI 只负责文本对话。
- 本章不实现多会话并行或对话列表管理；同一时刻只存在一个进行中的对话。
- 本章不实现语音、图片等多模态输入输出。
- 本章不实现应用内的模型切换菜单；模型与供应商的切换通过修改配置文件或启动参数完成。
- 本章不实现复杂的用户身份验证机制，仅通过 API 密钥完成模型供应商的认证。

## 验收标准

- **AC1** 启动应用后，TUI 正常渲染，界面中至少能看到对话展示区与用户输入区，无未处理异常退出（对应 F1）。
- **AC2** 在输入区输入任意文本并提交后，该文本以用户消息形式出现在对话展示区（对应 F2）。
- **AC3** 先询问「我的名字是什么」，再回答用户自己设定的名字，随后追问「我刚才问了你什么」，模型能基于同一会话的历史上下文正确回答（对应 F3）。
- **AC4** 提交一个需要多 token 回答的问题后，能在界面中观察到答案文字逐段出现，而不是停顿一段时间后一次性完整显示（对应 F4）。
- **AC5** 使用 OpenAI 兼容后端的配置可正常对话，且 Provider 抽象允许未来通过新增实现类扩展新后端（对应 F5）。
- **AC6** 当后端返回 `reasoning_content` 或类似思考字段时，TUI 先展示思考过程，再展示最终答案（对应 F6）。
- **AC7** 在一次完整对话后，TUI 底部状态栏显示累计的 prompt / completion / total token 数量（对应 F7）。
- **AC8** 在 YAML 配置文件中填写 `protocol`、`model`、`base_url`、`api_key` 后，应用启动能正确读取并使用该配置调用对应模型（对应 F8）。
- **AC9** 通过命令行参数指定另一个有效的 YAML 配置文件路径，应用使用该配置文件而不是默认路径（对应 F9）。
- **AC10** 当配置文件缺失、字段缺失或模型 API 返回错误时，TUI 中展示清晰的中文或英文错误提示，应用不崩溃（对应 N4）。
